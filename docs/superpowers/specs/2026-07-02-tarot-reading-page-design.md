# LUX — AI Tarot Reading Page — Design Spec

**Date:** 2026-07-02
**Status:** Approved (pending written-spec review)

## Summary

Add an interactive AI-powered tarot reading page to the existing LUX static site. A visitor
enters a question, draws a 3-card spread (Past / Present / Future), sees the cards revealed with
a flip animation, and receives a mystical, evocative reading generated by Groq (Llama 3.3 70B).
Ambient background music sets the mood. The page reuses the site's existing static-site
conventions and its Google Apps Script pattern — no build step, no new hosting, no new libraries.

## Guiding constraints

- **Simple & secure setup.** The Groq API key never reaches the browser. It lives in Google
  Apps Script **Script Properties** (server-side). The browser only talks to the existing
  Apps Script `/exec` URL.
- **No new third parties.** The only unavoidable external service is **Groq** (the AI provider).
  Everything else reuses what the project already has: Google Apps Script (proxy) and GitHub
  Pages (hosting). No npm, no bundler, no frameworks, no CDN libraries. Audio uses the built-in
  HTML5 `<audio>` element.
- **No build step.** Pure HTML/CSS/JS, matching the current site.

## Architecture & files

```
tarot-web/
├── reading.html              # New reading page
├── reading.css               # Reading-specific styles (reuses tokens/fonts from style.css)
├── reading.js                # UI orchestration, draw flow, proxy call, audio
├── deck.js                   # 78-card data model (generated): name, meanings, image path
├── design/assets/cards/      # Provided 78-card deck, "<index> <Name>.jpg" (0–77)
│   └── card-back.svg         # Generated card back (LUX celestial emblem)
└── design/assets/The Shadow of the Old Grove.mp3   # Ambient background track
```

The existing waitlist page (`index.html`, `style.css`, `main.js`) is **unchanged**. The reading
lives on a separate page so the launch/landing page carries no risk.

## Deck data model (`deck.js`)

The provided deck is named `<index> <Name>.jpg`, indices `0`–`77` in standard Rider-Waite order:

- `0`–`21` — Major Arcana (The Fool … The World)
- `22`–`35` — Wands (Ace … King)
- `36`–`49` — Cups
- `50`–`63` — Swords
- `64`–`77` — Pentacles

`deck.js` exports an array of 78 objects, one per index:

```js
{ id: 19, name: "The Sun", image: "design/assets/cards/19 The Sun.jpg",
  upright: "Joy, success, vitality, clarity.",
  reversed: "Temporary gloom, blocked joy, over-optimism." }
```

Meanings are curated standard Rider-Waite phrases (upright + reversed), generated as part of this
work. Filenames are derived from `id + " " + name`, matching the provided files exactly.

## Draw flow (`reading.js`)

1. User types a question (optional but encouraged) and clicks **Draw**.
2. First Draw click also starts the ambient audio (satisfies the browser autoplay gesture rule).
3. `drawSpread()` — a **pure function** — shuffles the 78-card indices (Fisher–Yates), takes the
   first 3 (guaranteed unique), and assigns each a random orientation (`upright` / `reversed`) and
   a position (`Past`, `Present`, `Future`).
4. Three face-down cards (the `card-back.svg`) render, then flip to reveal their images. Reversed
   cards are rotated 180°. Position labels appear beneath each.
5. The browser POSTs the drawn spread to the Apps Script proxy and shows a "the cards are
   speaking…" loading state.
6. The returned reading text renders below the spread.

`drawSpread()` returns a plain data structure and touches no DOM, so its logic (3 unique cards,
orientation randomness, position assignment) is verifiable in isolation.

## AI proxy (Google Apps Script)

The existing Apps Script web app gains a branch in `doPost(e)` that distinguishes reading
requests from waitlist-email requests (e.g., by a `type` field in the body).

**Request (browser → Apps Script):**
```json
{ "type": "reading",
  "question": "What should I focus on?",
  "cards": [
    { "name": "The Sun", "orientation": "upright", "position": "Past",
      "meaning": "Joy, success, vitality, clarity." }
  ] }
```

Apps Script builds a prompt and calls Groq via `UrlFetchApp.fetch()`:

- **Model:** `llama-3.3-70b-versatile` (Groq free tier).
- **System prompt:** mystical, evocative medium; ~2–3 short paragraphs; weave each card's
  position + orientation meaning into a single cohesive narrative that addresses the question.
- **Key:** read from `PropertiesService.getScriptProperties().getProperty('GROQ_API_KEY')`.

**Response (Apps Script → browser):** `{ "reading": "…text…" }`.

### CORS handling

The current waitlist call uses `mode: 'no-cors'`, which makes the response unreadable — fine for
fire-and-forget email, but the reading **must** read the text back. Fix: the browser sends a
**"simple" request** — POST with `Content-Type: text/plain` and **no custom headers** — which
skips the CORS preflight, so the Apps Script response body is readable via `fetch`. The body is
still JSON (parsed with `JSON.parse` on both ends); only the header is `text/plain`.

## Ambient audio

- Built-in `<audio loop>` element pointing at `The Shadow of the Old Grove.mp3`.
- Playback starts on the first user gesture (the Draw click), per browser autoplay policy.
- A subtle, always-visible **mute / unmute toggle** in a corner lets the user control sound.
  Preference persists via `localStorage`. Default: muted until the user opts in / draws.
- Respects the user: no autoplay with sound before interaction; toggle is keyboard-accessible.

## Error handling & fallback

- **Groq failure / 429 rate-limit / timeout / network error** → a graceful message with a
  **Retry** button.
- **Fallback reading:** even if Groq fails, the cards remain revealed with their curated
  upright/reversed meanings shown, so the experience never fully breaks.
- No streaming (Apps Script limitation) → single loading state, full text at once. Groq is fast
  (~1–3s), so this is acceptable.

## Known limitations (non-blocking for launch)

- **Public endpoint / quota abuse.** The `/exec` URL is public and consumes the free Groq quota;
  a script could spam it. Launch mitigations: client-side debounce and disabling the button
  during a request. Server-side rate-limiting (e.g., Apps Script `CacheService` throttle) is a
  documented follow-up, not launch-blocking.
- **Apps Script latency.** Cold starts and Google's per-account quotas apply. Fine for a
  low-traffic launch; revisit if traffic grows.

## Styling

`reading.css` reuses the site's tokens (`--parchment #F2E8D5`, `--oxblood #6B1A1A`,
`--gold #C9A84C`, `--ink #1A1412`) and fonts (Playfair Display + Inter). Layout is mobile-first,
consistent with the landing page. The card back (`card-back.svg`) uses the same palette.
Card flip + reveal respect `prefers-reduced-motion`.

## Testing

- **Draw logic:** `drawSpread()` is pure and checked in isolation — 3 unique cards, valid
  orientations, correct position assignment, no repeats across many runs.
- **UI, animation, audio, live Groq call:** manual QA. No test harness is added; the project is a
  vanilla static site and a build step would violate the no-third-parties / no-build constraint.

## Out of scope

- Card artwork (provided by user — done).
- Persisting/saving readings, accounts, history, sharing.
- Additional spreads beyond 3-card (single-card / Celtic Cross are future work).
- Server-side rate-limiting hardening (documented follow-up).
